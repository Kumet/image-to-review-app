# FastAPI + HTMX 複数画像アップロードアプリ 設計書

作成日: 2026-04-02
文書版: 1.0

==================================================
1. 目的
==================================================

本設計書は、FastAPI + HTMX を用いて、複数の写真をアップロードし、
サーバー側でダミー解析を実行し、その結果を画面に返す Web アプリケーションの
設計方針、構成、責務分離、非機能要件、ディレクトリ構成、開発ルール、
将来拡張方針を定義するものである。

現時点では返却内容は未確定であるため、解析処理はダミー実装とする。
ただし、将来的に OCR、画像分類、画像要約、レビュー生成、外部 AI 連携へ
安全に差し替えられる構造を前提とする。

==================================================
2. システム概要
==================================================

ユーザーはブラウザ上から複数画像をアップロードする。
FastAPI は multipart/form-data で画像群を受け取り、サーバー上に保存する。
保存後、ダミー解析サービスが画像ごとの簡易結果と全体サマリを生成する。
その結果を HTML partial として返却し、HTMX がページの一部を更新する。

基本方針:
- サーバーサイドレンダリング中心
- HTMX による部分更新
- JavaScript は最小限
- 画像保存と解析処理の責務を分離
- 将来の AI 化を見据えた疎結合設計
- 初期段階ではローカルファイル保存
- 初期段階では DB 未使用でも成立
- job_id 単位で処理を管理し、将来の DB 導入に備える

==================================================
3. ゴールと非ゴール
==================================================

3.1 ゴール
- 複数画像をアップロードできること
- アップロード直後に結果を画面へ反映できること
- 返却内容はダミーでも構造化されていること
- UI と処理ロジックが分離されていること
- 後から AI 処理へ差し替えやすいこと
- テストしやすい構造であること
- ローカル開発しやすいこと
- 将来的に S3 / DB / 非同期ワーカーへ拡張しやすいこと

3.2 非ゴール
- 初期版で本格的な AI 解析を実装すること
- 初期版で認証・課金・マルチテナントを実装すること
- 初期版で大規模分散処理を実装すること
- 初期版で本番向け監視基盤をフル実装すること

==================================================
4. ユースケース
==================================================

4.1 基本ユースケース
1. ユーザーがトップページへアクセスする
2. ユーザーが複数画像を選択する
3. ユーザーがアップロードボタンを押す
4. HTMX が multipart/form-data で POST する
5. FastAPI がファイルを検証し保存する
6. ダミー解析を実行する
7. 解析結果を HTML partial として返却する
8. HTMX が結果表示領域を書き換える

4.2 例外ユースケース
- 画像が 0 件
- 拡張子が不正
- ファイル数上限超過
- 画像サイズ上限超過
- 保存失敗
- ダミー解析失敗
- 想定外例外

6. 採用理由
==================================================

6.1 FastAPI
- 型ヒントベースで実装しやすい
- ファイルアップロード API を構築しやすい
- Jinja2 テンプレート併用が容易
- テストしやすい
- 将来 API と SSR の両立がしやすい

6.2 HTMX
- 部分更新が簡潔
- 小規模アプリで過剰な SPA 化を避けられる
- ファイルアップロードも相性が良い
- 初期実装の速度が高い

6.3 Jinja2
- サーバー側で HTML partial を返しやすい
- HTMX と相性がよい
- UI を Python 側データ構造と自然に結びつけられる

6.4 Pillow
- 画像のサイズ確認
- フォーマット確認
- 将来の簡易前処理に使える

7. 全体アーキテクチャ
==================================================

構成概要:

[Browser]
   |
   | GET /
   v
[FastAPI Pages Router] ----> [Jinja2 Templates]
   |
   | POST /uploads (HTMX multipart/form-data)
   v
[Uploads Router]
   |
   +--> [Validation]
   |
   +--> [File Service]
   |       |
   |       +--> save to local storage
   |
   +--> [Dummy Analyzer Service]
   |
   +--> [Result Service]
   |
   +--> return HTML partial
   v
[Browser HTMX partial swap]

将来拡張:

[Uploads Router]
   |
   +--> [Job Record]
   +--> [Async Queue]
   +--> [OCR / Vision / LLM]
   +--> [DB]
   +--> [Object Storage]

責務:
- FastAPI アプリの生成
- ルーター登録
- static mount

- 共通ミドルウェア登録

禁止事項:
- ビジネスロジックを直接書かない

- テンプレート組み立て処理を多く持たない

9.2 app/core
責務:
- 設定
- ログ設定
- 定数
- 例外定義

9.3 app/routes
責務:
- HTTP リクエストの入口
- リクエスト受理
- サービス呼び出し
- テンプレート返却 / partial 返却
- HTTP ステータス決定

禁止事項:
- 画像保存ロジック
- 解析ロジック
- 複雑な整形ロジック

9.4 app/services
責務:
- ビジネスロジック本体
- ファイル処理
- ダミー解析
- 表示用データの組み立て

9.5 app/schemas
責務:
- Pydantic による入出力型定義
- 内部データ構造の整備

9.6 app/templates
責務:
- 画面表示テンプレート
- partial の分離
- 表示に特化

9.7 app/static
責務:
- CSS / JS / 保存済みアップロードの配信

9.8 tests
責務:
- 回帰防止
- バリデーション確認
- partial レンダリング確認
- サービス単体試験

13. バリデーション設計
==================================================

13.1 ファイル数制限
- 0 件はエラー
- 最大件数を環境変数で制御
- 例: MAX_UPLOAD_FILES=10

13.2 拡張子制限
許可例:
- .jpg
- .jpeg
- .png
- .webp

13.3 MIME タイプ制限
許可例:
- image/jpeg
- image/png
- image/webp

13.4 サイズ制限
- 1 ファイルごとの最大サイズ
- リクエスト全体サイズも将来考慮

13.5 画像検証
- Pillow で画像として開けるか
- 壊れたファイルを弾く
- 必要に応じて width / height 取得

13.6 ファイル名の扱い
- 元ファイル名を保存名にそのまま使わない
- UUID + 正規化拡張子を使用する

15. ダミー解析設計
==================================================

15.1 目的
- UI フロー確立
- サービス分離
- 将来の AI 化に備えた契約面の固定

15.2 入力
- 保存済み UploadedImageInfo の配列

15.3 出力
- DummyAnalysisResult

15.4 ダミー処理例
各画像に対して:
- ラベルを疑似生成
- スコアをランダムまたは deterministic に生成
- コメントをテンプレートから生成

例:
- "サンプル画像として受理しました"
- "明るい印象のダミー結果です"
- "画像サイズに基づく仮スコア: 82"

15.5 重要方針
- ルーター内にダミー処理を書かない
- ダミーでも戻り値の構造を安定させる
- 将来 AI 化してもテンプレート側を壊さない

21. HTML / HTMX 設計方針
==================================================

- method="post"
- enctype 相当は hx-encoding で multipart/form-data
- input type=file multiple accept=image/*

21.2 HTMX 属性例
- hx-post="/uploads"
- hx-target="#result-area"
- hx-swap="innerHTML"
- hx-indicator="#loading"

21.3 推奨 UX
- アップロード中インジケータ表示
- 成功時に結果一覧表示
- 失敗時にエラーメッセージ表示
- 必要なら選択ファイル名プレビュー

36. 最終推奨
==================================================

このアプリは、最初から大規模な SPA や複雑なマイクロサービスにせず、
FastAPI + Jinja2 + HTMX を軸にしたシンプルなモノリスとして始めるのが最適である。

特に重要なのは以下の 5 点である。

1. ルーターを薄く保ち、処理を services に逃がすこと
2. ダミー解析でも戻り値を構造化しておくこと
3. job_id 単位で処理を管理すること
4. HTML partial 中心で HTMX に返すこと
5. 将来の AI / DB / 非同期化を見据えて、保存と解析を抽象化すること

この設計に従えば、初期実装は軽量でありながら、後から
画像解析アプリとして段階的に育てやすい構成になる。
